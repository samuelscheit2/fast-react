# Worker 016: Root Lockfile Sync

## Summary

Synchronized the root `package-lock.json` with the current workspace manifests using npm lockfile tooling:

```bash
npm install --package-lock-only --ignore-scripts
```

The stale metadata root cause was that worker-015 lowered the native binding package Node engine floor in `bindings/node/package.json` from the root workspace default `>=26.0.0` to `>=22.0.0`, but the root lockfile had not been regenerated afterward. npm updated the lockfile metadata for `packages["bindings/node"].engines.node` accordingly.

Worker-014's `packages/react/package.json` metadata was inspected too. The lockfile already matched the npm-recorded fields for `packages/react`: `name`, `version`, and `engines`. The richer package fields added there, such as `exports` and scripts, are not recorded in this lockfile's workspace package metadata, so npm produced no React lockfile diff.

## Stale Metadata Fixed

- `package-lock.json` -> `packages["bindings/node"].engines.node`
  - Before: `>=26.0.0`
  - Current manifest: `bindings/node/package.json` has `>=22.0.0`
  - After npm sync: `>=22.0.0`

No other workspace lock metadata mismatches were found for:

- root package metadata at `packages[""]`
- `packages["packages/react"]`
- `packages["tests/conformance"]`
- workspace link entries under `node_modules/@fast-react/*`
